# Google Search Engine

## Scale
8.5B searches/day (~100K QPS), index of 100B+ web pages, < 200ms end-to-end response time

## Core Challenges
Web crawling at scale, building and updating an inverted index, relevance ranking (PageRank + ML), autocomplete

## APIs (User-facing)
- GET /search?q=query&page=1 -> ranked results with titles, snippets, URLs
- GET /search?q=query&type=image|video|news
- GET /autocomplete?prefix=text -> top N search suggestions

## Core Components
- **Web Crawler (Googlebot)** - Distributed crawlers discover pages via BFS from seed URLs; respect robots.txt and crawl-delay
- **Content Store** - Raw HTML stored in distributed file system (Colossus/GFS); indexed by URL hash
- **Indexer** - Parses HTML; extracts text, links, metadata; builds forward and inverted indexes
- **Inverted Index** - term -> posting list [(doc_id, tf_score, positions, ...)] sharded across thousands of servers
- **Link Graph Store** - Stores all hyperlinks for PageRank computation
- **PageRank Engine** - Iterative MapReduce/Pregel job over the link graph; run offline; scores stored per URL
- **Query Processor** - Tokenize -> lookup index shards in parallel -> merge -> rank -> return top 10
- **Autocomplete Service** - Trie of popular queries with frequency weights; prefix lookup

## Deep Dive
- **Inverted Index sharding**: Sharded by term hash across N servers (each shard = subset of terms). Query fans out to all shards, each returns top K results for its terms, results merged and re-ranked by the query coordinator.
- **PageRank**: Iterative algorithm: PR(A) = (1-d) + d * sum(PR(T)/C(T)) for all pages T linking to A. Converges after ~50 iterations over the full web graph. Run offline on new crawl data using distributed graph processing.
- **Query serving latency**: 100-200ms budget. Index lookup is parallelized across shards (< 50ms). Re-ranking with ML models (BERT for understanding query intent). Result snippets generated by extracting relevant sentences containing query terms.
- **Freshness**: Caffeine (real-time indexing) processes newly crawled pages within seconds for news and trending topics, layered on top of the base full-web index rebuilt over weeks.

## Key Trade-offs
- Crawl freshness vs server load on crawled websites: respect crawl budget; prioritize high-authority, frequently-changing pages
- Index freshness vs index rebuild cost: real-time layer (Caffeine) for fresh content + full rebuild for the long-tail web
- Precision vs recall: PageRank + on-page signals + ML models balance these; continuous A/B testing of ranking changes
